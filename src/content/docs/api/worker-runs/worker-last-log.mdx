**Method:** `GET`

**Endpoint:** `/api/v2/workers/{workerId}/runs/last/log`

**Authentication:** Supports `Authorization: Bearer <YOUR_API_KEY>`, `api-key: <YOUR_API_KEY>`, and `?token=<YOUR_API_KEY>`. Prefer Bearer token.

## Try it

<ApiPlayground method="GET" path="/api/v2/workers/{workerId}/runs/last/log" />

## When to use this endpoint

Use this endpoint to read run logs for monitoring and troubleshooting.

## Identifier Notes

- `workerId` / `worker_id` accepts a Worker slug or a path encoded as `owner~name` from `owner/name`.

## Path Parameters

| Parameter | Required | Type | Description |
| --- | --- | --- | --- |
| `workerId` | Yes | `string` | Worker slug or path. You may paste `owner/name`; the playground sends it as `owner~name` for path values. |

## Request Example

```bash
curl -X GET "https://openapi.coreclaw.com/api/v2/workers/YOUR_WORKER_ID/runs/last/log" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

## Response Example

```json
{
  "code": 0,
  "data": {
    "all_logs_url": "https://storage.example.com/runs/01KKDXV2G26BT7NH4ZQR2R4NPZ/logs.txt",
    "list": [
      "Run started",
      "Run finished"
    ],
    "result_count": 1
  },
  "message": "success",
  "request_id": "req-123"
}
```

## Notes

- API v2 supports Bearer token, the legacy `api-key` header, and query token. Prefer Bearer token for new integrations.
- This is a Worker-scoped latest-run endpoint; `workerId` is required and the operation applies only to that Worker's latest run.
- `last` means the latest run in the current scope. Use a concrete `runId` endpoint when the target must be explicit.
